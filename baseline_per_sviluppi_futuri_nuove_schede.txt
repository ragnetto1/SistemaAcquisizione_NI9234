BASELINE TECNICA UFFICIALE PER SVILUPI FUTURI - MODULO NI9234
Progetto: SistemaAcquisizione_CDAQ
Data baseline: 2026-03-05
Owner baseline: Project Lead (Codex)
Target utilizzo: contesto completo per creare nuovi moduli (es. NI9229) mantenendo piena compatibilita con root orchestration, standalone e pipeline TDMS.

====================================================================
1) SCOPO DEL DOCUMENTO
====================================================================
Questo documento cristallizza la baseline tecnica del modulo NI9234 cosi come implementato nel repository corrente.
Obiettivo: fornire una specifica operativa dettagliata e riusabile per replicare architettura, comportamento e standard qualitativi su nuove schede NI senza regressioni.

La baseline copre:
- Architettura software e separazione responsabilita
- Contratti runtime tra UI, core acquisizione, writer, merge e root chassis
- Schema dati (workspace, sensor DB, TDMS metadata)
- Comportamenti UI e vincoli funzionali
- Pipeline FFT e canali calcolati
- Gestione errori, chiusura risorse e robustezza
- Delta analysis iniziale per adattamento NI9229

====================================================================
2) PERIMETRO E VINCOLI DI BASELINE
====================================================================
2.1 Perimetro modulo NI9234
- Modulo dedicato a NI9234 (4 canali analogici simultanei)
- UI PyQt5 con tab Canali, Canali Calcolati, Grafici (Chart, Instant, FFT)
- Acquisizione continua via NI-DAQmx callback Every N Samples
- Salvataggio TDMS segmentato in RAM + flush su file temporanei
- Merge finale TDMS con validazione e opzionale segmento FFT
- Compatibile sia standalone sia orchestrazione root (sync agent)

2.2 Vincoli architetturali da mantenere nei nuovi moduli
- Processo separato per modulo
- UI come orchestratore locale, AcquisitionManager come core real-time
- Thread separati per acquisizione, writer e worker FFT
- Root orchestration via ModuleSyncAgent e comandi sincronizzati
- Nessun crash UI causato da eccezioni callback
- Chiusura pulita di task DAQ, thread, timer, socket sync

4) MODELLO ARCHITETTURALE E THREADING
====================================================================
4.1 Thread model
- Main/UI thread:
  - Rendering UI, input utente, status, orchestrazione
  - Gestione sync commands via ModuleSyncAgent queue processing

- NI-DAQmx callback thread:
  - Esegue _on_every_n_samples
  - Legge blocco campioni, aggiorna last raw, enqueue verso writer
  - Emissione callback thread-safe verso UI (via segnali)

- Writer thread:
  - Consuma queue campioni
  - Applica gating sync_cutoff
  - Accumula in RAM fino a memory_limit
  - Flusha su segmenti TDMS temporanei

- FFT worker thread:
  - Riceve blocchi e costruisce finestra FFT su disco
  - Calcola spettro e invia risultato UI

4.2 Principi di isolamento
- UI non fa I/O pesante nei path critici di acquisizione
- Core acquisizione non dipende da widget UI
- Merge finale eseguito a stop con progress dialog
- Errori locali loggati o pop-up senza mandare in fault pipeline globale

5) DISCOVERY DEVICE E RATE MODEL
====================================================================
5.1 Discovery
- list_ni9234_devices_meta:
  - filtra device con product_type contenente 9234 o 4 AI channel
  - include is_simulated e chassis alias
- refresh_devices in UI:
  - costruisce combo dispositivi
  - supporta priorita da env chassis selection
  - supporta lock su device forzato via CDAQ_TARGET_DEVICE_NAME

5.2 Rate model
- _read_device_caps:
  - prova a leggere ai_max_single_chan_rate e ai_max_multi_chan_rate
  - fallback NI9234: single 51200 Hz, multi 51200 * num_channels
- _compute_per_channel_rate(device, n_channels)
  - n=1 => single_max
  - n>1 => agg_max / n
- set_user_rate_hz:
  - Fs utente opzionale (clamp a max hardware)
  - "Max" o empty => auto max

5.3 Semantica Fs in NI9234 baseline
- Fs rappresenta frequenza di campionamento per canale
- current_agg_rate_hz = Fs * numero_canali_attivi
- etichetta status mostra per-channel + aggregato + limiti modulo

7) PIPELINE ACQUISIZIONE LIVE
====================================================================
7.1 Trigger start acquisizione
- Cambio check canali abilitati
- Cambio Fs (se running, restart acquisizione)
- Cambio device (stop + reset plot)

7.2 start() core
- Crea task NI-DAQmx
- Aggiunge canali AI con range calcolato da limiti sensore convertiti in Volt
- Applica coupling/IEPE quando supportato
- Configura timing continuo
- Registra callback every_n_samples
- Avvia writer thread
- Start task

7.3 callback _on_every_n_samples
- Legge buffer float64 [nch x nsamples]
- Inizializza t0 con first block o sync anchor
- Aggiorna _last_raw per canale
- Applica zero (shift raw) + calibrazione a*x+b per UI channel value
- Enqueue blocco grezzo verso writer con global_start_index
- Emissione blocco instant e chart decimato

7.4 Robustezza callback
- try/except su tutte le callback UI
- errori non interrompono task
- early return se stop/closing/task non valido

11) SALVATAGGIO TDMS - PIPELINE DETTAGLIATA
====================================================================
11.1 Avvio salvataggio (_on_start_saving)
- Richiede acquisizione attiva
- Crea subdir temporanea timestampata
- Propaga label canali e formule al core
- Imposta memory_limit da spinRam
- clear_memory_buffer prima nuova sessione
- set_output_directory(subdir)
- set_base_filename
- set_recording(True)
- lock UI editing

11.2 Writer loop (acquisition.py)
- Se recording=False:
  - flush eventuale RAM
  - drena queue
- Se sync hold attivo (cutoff enorme sentinel): attende senza consumare queue
- Se recording=True:
  - consuma blocchi dalla queue
  - applica sync gate su start_index/end_index
  - accumula RAM fino a soglia
  - flush_memory quando memory_limit raggiunto

11.3 flush_memory
- Ordina blocchi per indice globale
- Calcola start_time assoluto da t0 + seg_start_idx/Fs
- Scrive file temporaneo .tmp e poi rename atomico .tdms
- Scrive segmenti TDMS con:
  - RootObject metadata waveform e sync
  - GroupObject("Acquisition")
  - Channel Time
  - Canali fisici in unita ingegneristica
  - Canali calcolati abilitati

11.4 Metadata TDMS principali (root)
- sample_rate
- channels
- start_index
- chunk_offset
- start_time_epoch_s
- segment_start_time_iso
- sync_anchor_epoch_s
- sync_write_cutoff_index
- wf_start_time
- wf_increment

11.5 Metadata canali fisici
- Description / description
- Unit / unit_string
- wf_start_time
- wf_increment
- physical_channel
- scale_a
- scale_b
- zero_applied

11.6 Metadata canali calcolati
- calculated_channel_id
- formula
- runtime_error (se presente)
- wf_start_time
- wf_increment

12) STOP, MERGE E VALIDAZIONE
====================================================================
12.1 stop_graceful
- Abilita recording per catturare residuo
- Attende breve drain
- Legge campioni residui da buffer hardware
- Enqueue blocchi finali
- Disabilita callback
- Chiude recording
- stop/close task
- stop/join writer

12.2 _on_stop UI
- stop_graceful + stop fallback
- disabilita FFT e cleanup worker files
- riabilita campi UI
- calcola final_path evitando overwrite (suffix datetime se necessario)
- istanzia TdmsMerger e merge folder temporanea
- mostra progress dialog con cancellazione
- se merge ok: elimina folder temporanea
- reset stato sync gate/anchor
- unlock table, uncheck canali

12.3 TdmsMerger comportamento
- scan segmenti .tdms ordinati
- prima pass: metadata, fs, t0, canali
- seconda pass streaming write su file finale
- supporto forced_wf_start_time
- append opzionale segmento FFT
- rename atomico tmp -> final
- validazione finale:
  - Time esiste e monotono
  - dt approx costante
  - lunghezze canali coerenti con Time

18) DELTA ANALYSIS INIZIALE PER NI9229 (USO BASELINE NI9234)
====================================================================
Nota: questo paragrafo e orientativo per pianificazione tecnica; i valori finali vanno validati contro specifica ufficiale NI e capability runtime via nidaqmx.

18.1 Vicinanza architetturale
NI9229 e piu vicino a NI9234 rispetto a moduli digitali/counter, perche:
- entrambi moduli analogici
- stessa famiglia di flusso: AI waveform, Fs per-canale, TDMS waveform
- riuso alto di UI, plot, merge, calcoli, sync orchestration

18.2 Differenze attese principali (da implementare)
- Front-end analogico differente (range, filtri, eventuale coupling supportato)
- Parametri min/max input e unita di default
- Capability rate reali e clamp da device caps
- Eventuali proprieta DAQmx non supportate su NI9229 (es. IEPE)

18.3 Strategia consigliata
- Fork logico NI9234 -> NI9229
- Conservare tutta la scaffolding orchestration/workspace/merge/formule
- Isolare differenze in:
  - acquisition.py: add channel config + timing/caps specifici
  - ui.py: colonne e default pertinenti alla scheda
  - gestione_risorse.py: naming canali se diverso

22) CRITERI DI ACCETTAZIONE BASELINE (GATE)
====================================================================
Una nuova scheda puo essere dichiarata "baseline-conforme" solo se:
1. Passa tutti i test della sezione 20 senza regressioni critiche.
2. Mantiene invariati i contratti sync (handlers, payload, snapshot).
3. Mantiene compatibilita workspace e formula engine.
4. Produce TDMS merge validato con metadata waveform coerenti.
5. Garantisce chiusura pulita risorse anche su error path.
